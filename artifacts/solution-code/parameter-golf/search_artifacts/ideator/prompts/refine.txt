## Problem
Your overarching goal is to create a `main.py` script that implements model training and evaluation to achieve the best possible performance (lowest Bits Per Byte) under strict constraints for the **OpenAI Parameter Golf** challenge.

## Challenge Overview
You are competing in the **Parameter Golf** challenge. The objective is to train the best language model that fits within a **16MB artifact constraint** and completes training in **under 10 minutes** on an 8xH100 GPU setup. You are optimizing for $L(N)$ - the lowest loss given a fixed number of parameters, unconstrained by data, compute bounds (aside from the 10 min limit), steps, or architecture.

## Script Interface
The `main.py` script must accept a `--mode` command-line argument that specifies the operation mode:
- `--mode train`: Train the model (training must stop before 10 minutes of wallclock time), then save the compressed trained model to `final_model.ptz`. Note: the total execution time of train mode may exceed 10 minutes due to post-training compression and saving.
- `--mode eval`: Load the compressed trained model from `final_model.ptz`, evaluate it on the validation set, and report `val_bpb` and `val_loss`. Time the eval loop with `time.perf_counter()` and, on rank 0 only, print a single line of the form `eval_time:<N>ms` (or `eval_time:<N>s` / `total_eval_time:<N>s`) once eval is complete. The runner parses this marker (case-insensitive, takes the largest match) and uses it — not subprocess wallclock — to enforce the 10-minute evaluation cap, so it must bound the actual evaluation work and exclude `torchrun` bring-up, model load, and `torch.compile` warm-up.

The script will be run within an Apptainer container using the following commands:
- Training: `torchrun --standalone --nproc_per_node=8 main.py --mode train`
- Evaluation: `torchrun --standalone --nproc_per_node=8 main.py --mode eval`
Note: Manual resource cleanup is not required following the completion of either the training or evaluation cycles.

## Strict Rules & Constraints

1. **Artifact Size Limit (< 16MB)**:
   - The submission artifact must be **under 16,000,000 bytes (decimal 16MB)**.
   - The artifact size is calculated as the sum of the **source code bytes** (your `main.py` script) plus the **compressed model bytes** (your `final_model.ptz` file).
   - You must find efficient ways to store and compress the model weights (e.g., using `zlib`, `bz2`, `lzma`, or applying techniques like int8/int6 Quantization-Aware Training or post-training quantization before saving).

2. **Compute Limits**:
   - **Training Time**: Training must stop in **under 10 minutes** of wallclock time on 8xH100s. The script must track wall-clock time from the start of training and stop training before the 10-minute limit is reached. Note: the total execution time of `--mode train` may exceed 10 minutes, as post-training steps (model compression and saving) are not subject to the 10-minute training limit.
   - **Evaluation Time**: The actual evaluation loop must complete in **under 10 minutes**. The runner enforces this against the `eval_time:` / `total_eval_time:` marker that `--mode eval` prints (see Script Interface above). If no marker is found it falls back to subprocess wallclock, which also charges `torchrun` startup, model load, and `torch.compile` warm-up (typically 60–180 s of overhead) — always emit the marker so only the real eval cost counts.
   - To iterate locally or on a single GPU, you may scale down hyperparameters appropriately, but the final submission must respect these limits scaled up.

3. **Evaluation Metrics**:
   - The primary evaluation metric is **tokenizer-agnostic bits per byte (bpb)** on the FineWeb validation set.
   - You **MUST NOT** cheat the validation metric. You cannot train on validation data, and you cannot test-time train on validation tokens before they have been predicted.
   - If you modify the tokenizer or dataset pipeline, you must ensure `val_bpb` is mathematically sound and correctly calculated based on raw bytes.
   - **How to calculate `val_bpb` correctly**:
     1. **Build byte-counting lookup tables from the SentencePiece tokenizer**: For each token in the vocabulary, precompute:
        - `base_bytes_lut[token_id]`: the number of UTF-8 bytes in the token's piece text **after stripping the leading `▁` (U+2581) marker**, or 1 for byte-fallback tokens. Control/unknown/unused tokens get 0.
        - `has_leading_space_lut[token_id]`: whether the token's piece starts with `▁` (U+2581).
        - `is_boundary_token_lut[token_id]`: True for control/unknown/unused tokens, False for normal tokens.
     2. **Accumulate bytes during evaluation**: For each target token in the validation set, the byte count is:
        `token_bytes = base_bytes_lut[target] + (has_leading_space_lut[target] & ~is_boundary_token_lut[prev_token])`
        The extra space byte for `▁` is only added when the previous token is NOT a boundary token (i.e., the space is part of a word boundary between two valid tokens).
     3. **Compute the final metric**: After accumulating `loss_sum` (total cross-entropy in nats), `token_count`, and `byte_count` across the entire validation set:
        `val_loss = loss_sum / token_count`
        `val_bpb = val_loss / ln(2) × (token_count / byte_count)`
        This converts nats-per-token → bits-per-token → bits-per-byte.
     4. **Common mistakes to avoid**:
        - Using `Ġ` (U+0120, the GPT-2/BPE convention) instead of `▁` (U+2581, the SentencePiece convention) to detect leading spaces. These are completely different characters.
        - Counting the raw UTF-8 bytes of the `▁` marker (3 bytes) instead of the 1 byte it represents (an ASCII space).
        - Forgetting to strip `▁` before counting the remaining piece's UTF-8 bytes.
        - Using `math.log2` or `/ 2` instead of `/ math.log(2)` (i.e., `/ ln(2)`) in the conversion.
        - Omitting the `(token_count / byte_count)` ratio, which converts from per-token to per-byte.
     5. **Tokenizers that re-encode text before SentencePiece (e.g. lossless caseops)**: The recipe in steps 1-3 assumes each SentencePiece piece's UTF-8 bytes (after stripping `▁`) equal the source-text bytes it represents. This assumption is **broken** for tokenizers like `fineweb_8192_bpe_lossless_caps_caseops_v1_reserved.model`, where text is bijectively lowercased and case operations (UPPER, Title, etc.) are encoded as reserved sentinel tokens by a pre-encoder BEFORE `sp.encode(...)`, and the inverse is applied at decode to recover the original cased text. Two consequences:
        - **(a)** Raw cased text CANNOT be fed directly to `sp.encode(...)` — apply the caseops pre-encoder first.
        - **(b)** The standard `base_bytes_lut[token_id]` recipe (UTF-8 bytes of each SentencePiece piece) is **wrong** because case-op sentinel tokens expand to multi-byte cased text on decode. Compute `val_bpb` against the actual decoded byte stream, or use a per-token byte-count sidecar produced at shard-prep time. Getting this wrong silently inflates or deflates `val_bpb` even though the cross-entropy is correct.

4. **Self-Contained Evaluation**:
   - No external downloads, training dataset access, or network calls are allowed during evaluation.
   - The 16MB submitted artifact (`main.py`, other Python files, and `final_model.ptz`) must be fully self-contained and reproducible.
   - Standard pip packages (e.g., FlashAttention, standard optimizers) are allowed as long as they do not sneak in extra capabilities or massively increase effective code size.
   - **Container-provided helpers** (do NOT reimplement these in `main.py` — `import` them): the training container ships `ngram_tilt.py` and `fused_expert_kernel.cpp` (precompiled into `/opt/ngram/libfused_ngram.so` at container build time), installed at `/opt/ngram_tilt.py` and `/opt/ngram/` with `PYTHONPATH=/opt`.
